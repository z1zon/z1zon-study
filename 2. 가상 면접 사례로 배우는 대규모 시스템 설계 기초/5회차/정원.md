# 분산 시스템을 위한 유일 ID 생성기 설계

## 분산 시스템에서 유일성이 보장되는 ID 생성 방법

### 1. 다중 마스터 복제

- DB의 auto_increment 기능을 활용 but, 현재 사용 중인 DB 개수 k만큼 증가시키는 방식
- 장점
  - DB 수를 증가함으로써 초당 생산 가능 ID 개수도 늘려 규모 확장성 문제 해소 가능
- 단점
  - 여러 데이터 센터에 걸쳐 규모를 늘리기 어려운 방식
  - ID의 유일성은 보장되지만 시간 흐름에 맞춰 커지도록 보장할 수 없음
  - 서버를 추가하거나 삭제할 때 정상 동작이 어려움

### 2. UUID

- 컴퓨터에 저장되는 정보를 유일하게 식별하기 위한 128비트 자리수
- 장점
  - 서버 사이의 조율을 할 필요가 없기 때문에 동기화 이슈가 없음
  - 규모의 확장성도 보장되는 구조
- 단점
  - ID가 64비트를 넘어섬
  - ID를 시간 순으로 정렬 불가능
  - ID에 숫자가 아닌 값이 포함될 수 있음

### 3. 티켓 서버

- auto_increment 기능을 갖춘 중앙 집중형 DB 서버인 티켓 서버를 하나만 사용하는 방식
- 장점
  - 유일성 보장 및 숫자로만 구성된 ID값을 쉽게 생성 가능
  - 구현이 쉽고 중소 어플리케이션에 적합한 방식
- 단점
  - 티켓서버가 SPOF(Single-Point-Of-Failure)가 됨
  - 티켓서버를 여러 대 운영하면 신경서야 할 데이터 동기화 문제가 발생함

### 4. 트위터 스노우 플레이크 접근법

- |-|------------------------------------------|-----|-----|------------|
  sign timestamp DB센터 서버 일련번호
  ID ID

- sign (1 bit): 음수/양수 구별
- timestamp (42 bits): 기원 시각 이후로 몇 ms 경과했는지 나타내는 값
- DB Center ID (5 bits): 2^5=32개의 데이터센터 지원 가능한 값
- Server ID (5 bits): 데이터센터 당 2^5=32개 서버를 사용 가능한 값
- Serial Number (12 bits): 각 서버에서 ID를 생성할 때마다 일련 번호를 1만큼 증가시킴, 1ms가 경과할 때마다 0으로 초기화됨

### 생각해봐야 할 다른 문제들

1. 시계 동기화

- 하나의 서버가 여러 코어에서 실행되거나 서버가 물리적으로 독립된 여러 장비에서 실행되는 경우 서버마다 다른 시계를 사용하는 문제

2. 각 절(section)의 길이 최적화

- 동시성이 낮고 수명이 긴 애플리케이션이라면 일련번호 절의 길이를 줄이고 타임스템프 절의 길이를 늘리는 방식으로 최적화 도모
